# Running the CFR experiments on a fresh machine

End-to-end recipe for the three TODO blocks in
`vllm-paper/sections/evaluation.tex` (lines 36 / 45 / 63):

1. `run_validation_cfr.sh`       — controller-validation (TODO 36)
2. `run_adaptive_selector_cfr.sh` — MB / EB / ADA selector (TODO 63)
3. `run_grid_search_cfr.sh`      — `(B, N)` grid for v1 vs EB(k̂) (TODO 45)

All three scripts share the helpers in `common_cfr.sh`: GPU auto-detection,
calibration resolution, output paths under
`outputs/<exp>/<GPU_TAG>_<MODEL_SHORT>/`. They each take one positional arg
`MAX_GPUS` and a few env-var overrides — see script headers for the full list.

## 0. Prereqs

```bash
cd /path/to/vllm

# venv: pd_exp/common.sh's init_experiment_env auto-sources `<repo>/.venv/bin/activate`
.venv/bin/vllm --version       # must work

# calibration files (per model; per GPU type if you have measured them):
ls pd_exp/outputs/pd_calibration_Qwen3-8B.json
ls pd_exp/outputs/pd_calibration_Qwen3-30B-A3B.json
```

If a calibration file is missing or was generated on a different GPU, refresh:

```bash
.venv/bin/python -m vllm.v1.core.sched.calibration \
    --model Qwen/Qwen3-8B \
    --output pd_exp/outputs/pd_calibration_Qwen3-8B_<GPU_TAG>.json
```

`common_cfr.sh::resolve_calibration` looks for the per-GPU file first
(`pd_calibration_<MODEL>_<GPU_TAG>.json`) and falls back to the legacy
`pd_calibration_<MODEL>.json` with a warning.

`<GPU_TAG>` is generated by `detect_gpu_name`:
H200 / H100 / B300 / B200 / RTXPRO6000 / L40S / A6000 / ...

## 1. Run order (short → long)

Pass your GPU count as the positional arg (default is 3-4 if omitted):

```bash
cd /path/to/vllm/pd_exp/syn_cfr
mkdir -p outputs/_runner_logs

# === Qwen3-8B ===
MODEL=Qwen/Qwen3-8B ./run_validation_cfr.sh 8 \
    > outputs/_runner_logs/validation_qwen3-8b.log 2>&1
MODEL=Qwen/Qwen3-8B ./run_adaptive_selector_cfr.sh 8 \
    > outputs/_runner_logs/selector_qwen3-8b.log 2>&1
MODEL=Qwen/Qwen3-8B ./run_grid_search_cfr.sh 8 \
    > outputs/_runner_logs/grid_qwen3-8b.log 2>&1

# === Qwen3-30B-A3B ===
MODEL=Qwen/Qwen3-30B-A3B ./run_validation_cfr.sh 8 \
    > outputs/_runner_logs/validation_qwen3-30b.log 2>&1
MODEL=Qwen/Qwen3-30B-A3B ./run_adaptive_selector_cfr.sh 8 \
    > outputs/_runner_logs/selector_qwen3-30b.log 2>&1
MODEL=Qwen/Qwen3-30B-A3B ./run_grid_search_cfr.sh 8 \
    > outputs/_runner_logs/grid_qwen3-30b.log 2>&1
```

Each script writes a `progress.txt` under its output directory with one
`OK|<sched>|<scenario>|<bs>|<tb>` (or `FAIL|...`) line per experiment, plus
an `experiment_config.json` capturing the full config.

`SKIP_EXISTING=1` is on by default — if a `bench_*.json` already exists the
cell is skipped, so a re-run resumes from where it left off.

## 2. Analyze

```bash
cd /path/to/vllm/pd_exp/syn_cfr

# Per (experiment, model) — adjust <GPU_TAG> to match your hardware
.venv/bin/python analyze_cfr_validation.py \
    outputs/controller_validation/<GPU_TAG>_Qwen3-8B
.venv/bin/python analyze_cfr_selector.py \
    outputs/adaptive_selector/<GPU_TAG>_Qwen3-8B
.venv/bin/python analyze_cfr_e2e.py \
    outputs/e2e_grid_search/<GPU_TAG>_Qwen3-8B
```

Each writes `summary.csv`, an `optimal_*` CSV, a `*_table.tex`, and (for
the grid) per-scenario throughput heatmaps.

## 3. Run in the background, monitor progress

```bash
nohup ./run_grid_search_cfr.sh 8 > outputs/_runner_logs/grid_qwen3-8b.log 2>&1 &

# progress
PROG=outputs/e2e_grid_search/<GPU_TAG>_Qwen3-8B/progress.txt
watch -n 30 "wc -l $PROG; tail -3 $PROG"

# any failures
grep "^FAIL" $PROG
```

## 4. Useful env overrides

```bash
# Smoke test (one cell)
SCENARIOS="balanced" BS_VALUES="1024" TB_VALUES="14336" \
    NUM_PROMPTS=1000 MODEL=Qwen/Qwen3-8B ./run_grid_search_cfr.sh 1

# Pin to specific GPUs (skips auto-selection)
GPUS=0,1,2,3 MODEL=Qwen/Qwen3-8B ./run_grid_search_cfr.sh

# Force re-run of completed cells
SKIP_EXISTING=0 MODEL=Qwen/Qwen3-8B ./run_grid_search_cfr.sh 8

# Tighter / looser OOM tolerance for the controller (default 0.01 = 1%)
VLLM_PD_OOM_TOLERANCE=0.005 MODEL=Qwen/Qwen3-8B ./run_validation_cfr.sh 8

# Reduce grid axes
BS_VALUES="256 512 1024" \
TB_VALUES="8192 14336 18432" \
    MODEL=Qwen/Qwen3-30B-A3B ./run_grid_search_cfr.sh 8
```

Common knobs (all have sensible defaults — see script headers):

- `MODEL` — HF model id, default `Qwen/Qwen3-8B`
- `NUM_PROMPTS=4000`, `MAX_CONCURRENCY=2048` — paper-default load
- `RANDOM_RANGE_RATIO=0.5` — input length spread
  (`Unif[0.5·E[L], 1.5·E[L]]`)
- `SCHEDULERS` — default `"v1 eb_khat"` for grid; `"v1 eb_khat ada"` for
  selector
- `BS_VALUES`, `TB_VALUES` — grid axes
- `SKIP_EXISTING` — `1` to resume, `0` to overwrite
- `VLLM_PD_AUTO_COMPUTE_N=1` — memory-safe N̂ from Proposition memory;
  set to `0` to pin N to `--max-num-seqs`
- `VLLM_PD_OOM_TOLERANCE=0.01` — ε for the memory-safe N̂

For the adaptive selector, an offline kernel sweep gives a more accurate
diagnostic Δ(N):

```bash
export VLLM_PD_BETA_MB_E=...    # f(\bar r)
export VLLM_PD_ALPHA_MB=...
export VLLM_PD_CP_COST_A=...    # f(r) = a + b·r + c·r^2
export VLLM_PD_CP_COST_B=...
export VLLM_PD_CP_COST_C=...
./run_adaptive_selector_cfr.sh 8
```

Without these, the auto mode falls back to a conservative LHS=0 form and
tends to bias toward MB. (See `run_adaptive_selector_cfr.sh` header.)

## 5. Reference timings (RTX PRO 6000 Blackwell × 8, 96 GB each)

| Stage | Qwen3-8B | Qwen3-30B-A3B |
| --- | ---: | ---: |
| validation (6 cells) | ~7 min | ~14 min |
| adaptive selector (9 cells) | ~12 min | ~22 min |
| grid search (180 cells) | ~1.5 h | ~7-8 h |

H200 should be faster across the board (higher bandwidth ⇒ EB advantage
shrinks, MB advantage grows; fits with the bandwidth-driven crossover the
paper predicts). 30B-A3B at BS=2048 fits on a single 96 GB card without
TP — H200 (141 GB) has more headroom.

## 6. Output directory layout

```
outputs/
├── _runner_logs/                              # nohup stdouts
├── controller_validation/<GPU>_<MODEL>/
│   ├── experiment_config.json
│   ├── progress.txt
│   ├── validation_summary.csv
│   ├── validation_table.tex
│   └── <scenario>_in<L>_out<O>/
│       ├── bench_v1.json   bench_eb_khat.json
│       └── v1_stats.json   eb_khat_stats.json   (CFR update history)
├── adaptive_selector/<GPU>_<MODEL>/
│   ├── selector_summary.csv
│   └── selector_table.tex
└── e2e_grid_search/<GPU>_<MODEL>/
    ├── summary.csv
    ├── optimal_per_scheduler.csv
    ├── optimal_table.tex
    ├── heatmap_<scenario>_total_token_throughput.png
    └── tb<TB>/bs<BS>/<scenario>_in<L>_out<O>/bench_<sched>.json
```

The `*_stats.json` files include the CFR update history (per-update
snapshot of p̂_0, μ̂_L, θ_0, k̂, N̂, Δ(N), cumulative OOM count) used by
`analyze_cfr_validation.py`.
